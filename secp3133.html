Assignment 1: Academic Writing

Format: Academic Paper / Report

Technical Writing in Data Architecture: Databricks Lakehouse vs. Hadoop MapReduce

This academic paper presents a comparative analysis between the traditional Hadoop MapReduce framework and the modern Databricks Lakehouse platform for big data analytics. It evaluates the architectural shift from tightly coupled, disk-based batch processing to decoupled, in-memory computation utilizing Apache Spark and Delta Lake. The research highlights the critical trade-offs in processing speed, scalability, and ACID-compliant data reliability, supported by a real-world case study on high-frequency stock market analytics.

Throughout the development of this report, a strong emphasis was placed on academic integrity and rigorous citation standards. All technical claims regarding data architecture layers and performance metrics were strictly grounded in peer-reviewed, Scopus-indexed literature to ensure the highest level of factual accuracy and professional technical writing.

Project 1: Optimizing High-Performance Data Processing for Large-Scale Web Crawlers

Malay Mail Data Pipeline Optimization | April 2026 - May 2026

Automated Web Scraper & Polars Pipeline

Project Overview: Developed an automated web scraper and high-performance data pipeline specifically engineered to handle over 120,000 articles from the Malay Mail.

  • Extraction: Utilized a Python-based architecture integrating Playwright to execute robust, dynamic web extraction while bypassing strict server pagination limits.
  • Processing: Replaced standard single-threaded data manipulation tools with Polars to leverage its lazy-evaluation engine and Apache Arrow memory format.
  • Outcome: The optimization techniques successfully achieved a 358% increase in overall data processing throughput compared to the Pandas baseline.

Reflection: The most significant challenge was overcoming memory exhaustion and connection crashes during the deep-pagination scraping phase, which required redesigning the scraper into a resilient "Append-Only" batching engine. Transitioning from Pandas to Polars completely shifted my perspective on large-scale data engineering; witnessing how multi-threading and strict schema typing could bypass single-core bottlenecks to process 120,000+ rows in under 7 seconds proved that framework selection is just as critical as hardware.

Project 2: Real-Time Sentiment Analysis using Apache Spark and Kafka

Format: Real-Time Streaming Pipeline | Status: In Progress

Streaming Data & Sentiment Classification on Malaysian Google Maps Reviews

Project Overview: Designed a real-time sentiment analysis pipeline using Apache Kafka to ingest Malaysian Google Maps review data, processed through a CNN-based NLP model to classify text as Positive, Neutral, or Negative. The classified results are stored in Elasticsearch and visualized through an interactive Kibana dashboard, with performance compared between batch and streaming processing modes.

  • Ingestion: Apache Kafka was configured with a single broker and a dedicated topic (sentiment-stream) to simulate real-time data ingestion, with a Python producer streaming review records row-by-row at controlled intervals.
  • Processing: A 1D Convolutional Neural Network (CNN) trained on the cleaned review dataset was integrated into both a streaming consumer (reading from Kafka) and a batch consumer (using local Spark session), applying tokenization and padded sequence inference to classify each review's sentiment.
  • Outcome: Classified results are indexed into Elasticsearch across three indices — sentiment_results (streaming), sentiment_batch (batch), and performance_metrics — and visualized in Kibana with sentiment distribution charts, trend over time, and batch vs. streaming performance comparisons. (Dashboard currently in progress.)

Reflection: Working with real-time data streams introduced challenges that batch processing does not — particularly around pipeline coordination, where Kafka, Elasticsearch, and the CNN consumer must all be running simultaneously for the system to function correctly. Unlike batch processing where data is static and errors can be fixed before rerunning, streaming requires careful handling of message ordering, consumer timeouts, and partial batch writes. Setting up the infrastructure locally using Docker also surfaced hardware constraints around memory and disk space that would not appear in a cloud environment.
